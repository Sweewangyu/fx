# 基于 SFT+DPO 与数据/Schema 清洗的 CMEIA 医疗 Schema 约束三元组抽取报告

## 1. 项目主线

这个项目的真实推进逻辑是：

```text
拿到 CMEIA 医疗三元组数据
-> 构造 Schema 约束 SFT 指令数据
-> 训练第一版 SFT 小模型
-> 分析 badcase，发现过度抽取和关系混淆
-> 用负 schema 采样和 Hard Negative 缓解过抽取
-> 用 DPO 处理偏好输出，如“儿童肥胖”“持续上腹阵痛”
-> 发现精确率仍然不高，判断瓶颈不只在训练方法，而在数据质量和 schema 质量
-> 用反向合成清洗 text-triple 对齐质量
-> 用本体建模方法清洗和重构 schema
-> 用清洗后的数据和 schema 重新训练并评测下游 Graph RAG
```

这个项目最重要的点不是“堆了很多后训练方法”，而是我在迭代过程中逐步判断问题来源：

- 第一阶段问题是模型行为问题：过度抽取、关系混淆、实体边界偏好不好。
- 第二阶段问题是数据问题：原始文本和 gold 三元组不完全对齐。
- 第三阶段问题是 schema 问题：公开数据集 schema 粒度不统一、关系语义重叠、头尾类型约束弱。

所以最终方案是：**SFT/DPO 解决模型对齐问题，反向合成清洗解决数据质量问题，本体建模解决 schema 质量问题**。

最终结果：

| 阶段 | Precision | Recall | F1 | 关键动作 |
| --- | ---: | ---: | ---: | --- |
| 原始 SFT | 29.8 | 42.6 | 35.1 | CMEIA 指令数据冷启动 |
| + 负 schema 采样 | 37.6 | 47.8 | 42.1 | 抑制过度抽取 |
| + 关系簇 Hard Negative | 41.9 | 50.6 | 45.8 | 强化近义关系边界 |
| + 实体边界 DPO | 44.7 | 51.8 | 48.0 | 对齐医学实体边界和修饰语偏好 |
| + 反向合成数据清洗 | 53.4 | 58.7 | 55.9 | 只清洗训练集中的 text-triple 弱对齐样本 |
| + 本体建模 Schema 清洗 | 59.8 | 65.2 | 62.4 | 重构 relation 定义、候选 schema 和 head-tail 类型约束 |

## 2. 项目背景

在 Storage for AI 场景中，医院客户希望构建医疗 Graph RAG。原型 pipeline 使用 DeepSeek-R1 对医疗文档做三元组抽取，但客户侧存储设备没有足够算力长期部署大模型。因此目标是训练一个本地 7B 级小模型，在固定医疗 schema 约束下完成三元组抽取，为图谱构建和 Graph RAG 提供结构化输入。

业务价值：

- 用小模型替代大模型，降低本地部署成本。
- 通过 schema 约束控制图谱关系集合，避免开放抽取导致关系膨胀。
- 提高医疗图谱边质量，降低 Graph RAG 检索错误证据的概率。

## 3. 数据与任务定义

数据来源是 CMEIA 医疗三元组数据。每条样本包含：

- 一段医疗文本。
- 若干 gold 三元组。
- 一个全局医疗关系 schema。

任务输入：

```json
{
  "text": "儿童肥胖的发生与多吃少动、遗传因素有关。",
  "schema": ["病因", "发病机制", "临床表现", "治疗方法", "并发症"]
}
```

期望输出：

```json
{
  "病因": [
    ["儿童肥胖", "病因", "多吃少动"],
    ["儿童肥胖", "病因", "遗传因素"]
  ],
  "发病机制": [],
  "临床表现": [],
  "治疗方法": [],
  "并发症": []
}
```

这个格式有两个约束：

- 只能在候选 schema 内抽取，不能开放式生成关系。
- 候选 schema 没有证据时必须输出空数组。

评测口径：

- 主指标使用固定原始 Test 集，不清洗 Test 集，也不把 Test badcase 回流训练。
- 三元组匹配采用严格匹配：`head`、`relation`、`tail` 三者都完全一致才算正确。
- 实体边界严格计分，`儿童肥胖` 抽成 `肥胖`、`持续上腹阵痛` 抽成 `上腹阵痛` 都算错误。
- JSON 格式错误样本按空输出处理，避免格式失败被忽略。

数据规模：

| 数据 | 数量 | 用途 |
| --- | ---: | --- |
| CMEIA 原始样本 | 约 1.4w | 基础三元组抽取数据 |
| SFT 基础指令样本 | 1.4w | 原文级 train/dev/test 切分 |
| 负 schema 扩增样本 | 约 8.6w | 仅训练集扩增，每条文本多轮采样负关系 |
| Hard Negative 样本 | 1.9w | 近义关系和同 head-tail 不同 relation 对比 |
| DPO 偏好对 | 2.1k | 实体边界、修饰语保留、关系混淆 badcase |
| 固定 Dev 集 | 1.4k | 阈值和超参选择 |
| 固定 Test 集 | 1.4k | 主结果评测 |

为什么必须做 schema 约束：

- 医疗 Graph RAG 需要可控关系集合，否则图谱关系会快速膨胀。
- 开放式抽取会产生大量同义关系，如“导致”“引起”“诱发”“相关因素”，后续归并成本很高。
- 客户场景更关注图谱稳定性，而不是开放抽取的表面召回。

## 4. 第一阶段：SFT 数据构造与冷启动

### 4.1 SFT 数据构造

第一步是把 CMEIA 原始三元组数据改造成指令微调格式：

```text
Instruction:
你是医疗知识图谱三元组抽取专家。请根据原文和候选 schema 抽取三元组。没有证据的关系输出空数组。

Input:
文本 + 候选 schema

Output:
按 schema 分组的 JSON 三元组
```

候选 schema 不是把全量几十个关系全塞进 prompt，而是每条样本保留正关系，再采样若干负关系，把候选关系控制在 5 个左右。

训练配置：

| 项目 | 配置 |
| --- | --- |
| 基座模型 | Qwen2.5-7B-Instruct |
| 微调方式 | LoRA |
| rank / alpha | 64 / 128 |
| learning rate | 2e-4 |
| epoch | 3 |
| max length | 4096 |
| 输出格式 | 按 schema 分组的 JSON |

为什么不放全量 schema：

- 全量 schema 太长，训练和推理成本高。
- 对小模型来说，长 schema 会稀释文本证据信号。
- 大部分关系和当前文本无关，会制造无效上下文。

为什么要加入负 schema：

- 只训练正例会让模型形成“看到实体就抽”的倾向。
- 负 schema 告诉模型：候选关系出现在 prompt 中，不代表一定要抽。
- 这直接对应后续发现的过度抽取问题。

### 4.2 第一版 SFT 结果

第一版 SFT 后，模型格式基本稳定，但 Precision 只有 29.8，Recall 约 42.6。badcase 中最明显的问题是**过度抽取**。

典型错误：

| 原文 | 候选 schema | 模型错误输出 | 正确输出 |
| --- | --- | --- | --- |
| “2 型糖尿病患者应控制饮食并定期监测血糖。” | 治疗方法、并发症、检查项目 | `("2型糖尿病","并发症","肾病")` | `("2型糖尿病","治疗方法","控制饮食")`、`("2型糖尿病","检查项目","监测血糖")` |

错误原因：

- 模型学到了医学常识，但没有严格遵守“原文证据”。
- 训练集中负 schema 不够，模型没有充分学会输出空。
- 医学实体和医学关系共现时，小模型倾向于补全合理但无证据的关系。

## 5. 第二阶段：Badcase 驱动的模型对齐

### 5.1 过度抽取：负 schema 扩增

第一版 badcase 显示模型看到候选关系后容易过抽取，所以我把每条文本多次采样负 schema，相当于让同一文本在不同负关系组合下重复训练。

做法：

- 原来不足 5 个 schema 时补齐到 5 个。
- 后来对每条文本进行多轮负 schema 采样。
- 让大量候选关系的 gold 输出为空。

负 schema 采样策略：

| 采样策略 | 说明 | 作用 |
| --- | --- | --- |
| 随机负关系 | 从全局 relation schema 中随机采样无证据关系 | 提高输出空数组能力 |
| 同实体类型负关系 | 采样 head/tail 类型相近但无证据的关系 | 避免模型只靠实体类型判断 |
| 高频混淆负关系 | 优先采样病因/发病机制、治疗方法/药物治疗等易混关系 | 提升 relation boundary |
| 多轮负采样 | 同一文本构造多种候选 schema 组合 | 降低“看到候选关系就抽”的先验 |

为什么用负 schema 扩增：

- 过度抽取不是格式问题，而是拒抽能力不足。
- 只有看到足够多“候选关系存在但无证据”的训练样本，模型才会学会输出空。

为什么不用 prompt 直接写“不要过度抽取”：

- Prompt 对大模型可能有效，但对 7B 小模型不稳定。
- 过度抽取是模型行为偏好问题，需要通过训练分布改变。

结果：

| 方法 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: |
| 原始 SFT | 29.8 | 42.6 | 35.1 |
| + 负 schema 扩增 | 37.6 | 47.8 | 42.1 |

错误分布也发生了变化：

| 错误类型 | 原始 SFT | + 负 schema 扩增 |
| --- | ---: | ---: |
| 无证据过抽取 | 38.4 | 26.7 |
| 关系混淆 | 24.1 | 23.5 |
| 实体边界错误 | 18.3 | 17.9 |
| 漏抽 | 19.2 | 31.9 |

负 schema 扩增主要压低过抽取，但会带来一定漏抽上升，所以后面需要 Hard Negative 和 DPO 继续处理关系边界与实体边界。

### 5.2 关系混淆：Hard Negative

第二类 badcase 是近义关系混淆，尤其是：

- 病因 vs 发病机制。
- 症状 vs 临床表现。
- 治疗方法 vs 药物治疗。
- 并发症 vs 相关疾病。

典型错误：

| 原文 | 候选 schema | 模型错误输出 | 正确输出 |
| --- | --- | --- | --- |
| “高钾血症常见于肾小球滤过功能降低，导致钾排泄减少。” | 病因、发病机制、危险因素 | `("高钾血症","发病机制","肾小球滤过功能降低")` | `("高钾血症","病因","肾小球滤过功能降低")` |

Hard Negative 构造方式：

- 统计同 head-tail、不同 relation 的样本。
- 根据关系名、描述和 head/tail 类型构造关系簇。
- 训练时优先把易混淆关系放到同一个候选 schema 中。

关系簇示例：

| 关系簇 | 容易混淆的关系 | 区分依据 |
| --- | --- | --- |
| 病因类 | 病因、诱因、危险因素、发病机制 | 是否是直接原因、风险因素还是机制过程 |
| 表现类 | 临床表现、症状、体征 | 是否是主观症状、客观体征或泛化表现 |
| 治疗类 | 治疗方法、药物治疗、手术治疗 | tail 是否为药物、手术或泛化治疗方案 |
| 疾病关系类 | 并发症、相关疾病、鉴别诊断 | 是否存在并发、相关或诊断排除关系 |

为什么用 Hard Negative：

- 随机负关系太简单，模型很容易判断为空。
- 真正难的是在近义关系里选对一个。
- Hard Negative 强迫模型学习关系边界，而不是只学会有没有关系。

为什么不用后处理规则：

- 后处理只能过滤明显不合法的关系，不能提升模型内部的关系选择能力。
- 关系混淆发生在生成阶段，必须通过训练让模型学会区分。

Hard Negative 单独带来的结果：

| 方法 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: |
| + 负 schema 扩增 | 37.6 | 47.8 | 42.1 |
| + 关系簇 Hard Negative | 41.9 | 50.6 | 45.8 |

### 5.3 偏好输出：DPO 处理医学实体边界

第三类 badcase 是输出偏好不符合医学场景，尤其是实体边界和修饰语保留。

典型 badcase：

| 类型 | 原文 | 模型输出 rejected | 期望输出 chosen |
| --- | --- | --- | --- |
| head 边界过短 | “儿童肥胖的发生与多吃少动有关。” | `("肥胖","病因","多吃少动")` | `("儿童肥胖","病因","多吃少动")` |
| tail 修饰语丢失 | “胰腺结石患者常表现为持续上腹阵痛。” | `("胰腺结石","症状","上腹阵痛")` | `("胰腺结石","症状","持续上腹阵痛")` |

为什么用 DPO：

- 这类错误不是完全不会抽，而是“哪个输出更好”的偏好问题。
- “肥胖”和“儿童肥胖”都有一定语义相关性，但医学图谱里后者更准确。
- SFT 只能告诉模型 chosen 是什么，不能显式告诉模型 rejected 为什么不好。
- DPO 直接优化 chosen 相对 rejected 的偏好，更适合边界和修饰语对齐。

为什么不用继续加 SFT：

- 继续 SFT 会把 chosen 当作普通标签，但没有利用 rejected 的对比信息。
- DPO 能让模型明确降低错误边界输出的概率。

DPO 数据构造：

| 偏好类型 | chosen | rejected 来源 |
| --- | --- | --- |
| head 边界 | 保留疾病亚型、年龄、人群修饰语 | 第一版模型输出的过短 head |
| tail 边界 | 保留程度、持续时间、部位修饰语 | 第一版模型输出的过短 tail |
| 关系偏好 | 选择 schema 定义下更精确的 relation | Hard Negative 后仍混淆的模型输出 |
| 空输出偏好 | 无证据关系输出空数组 | 模型根据医学常识补全的 unsupported triple |

DPO 配置：

| 项目 | 配置 |
| --- | --- |
| preference pair | 2.1k |
| beta | 0.1 |
| learning rate | 5e-6 |
| epoch | 1 |
| reference model | Hard Negative SFT checkpoint |

结果：

| 方法 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: |
| + 负 schema 扩增 | 37.6 | 47.8 | 42.1 |
| + 关系簇 Hard Negative | 41.9 | 50.6 | 45.8 |
| + 实体边界 DPO | 44.7 | 51.8 | 48.0 |

DPO 对全量 F1 的提升不算特别大，但对边界类指标更明显：

| 指标 | DPO 前 | DPO 后 |
| --- | ---: | ---: |
| Head Boundary Exact | 63.2 | 70.5 |
| Tail Modifier Retention | 58.7 | 67.4 |
| Unsupported Triple Rate | 21.8 | 18.9 |

## 6. 第三阶段：发现瓶颈转向数据清洗

完成负 schema、Hard Negative 和 DPO 后，Precision 仍然不到 50。继续看 badcase 后，我判断瓶颈不只在模型，而在原始数据质量。

主要现象：

- 有些文本很长，但 gold 三元组很少。
- 文本中存在明显可抽关系，但标注没有覆盖。
- 部分 gold 三元组和原文证据关系弱。
- schema 的关系粒度不统一，近义关系边界不清晰。

这时如果继续堆训练技巧，收益会变低，因为模型会被噪声标注牵着走。因此后续重点转向两件事：

- 清洗 text-triple 对齐质量。
- 清洗和重构 schema。

## 7. 第四阶段：反向合成清洗 text-triple 对齐

### 7.1 方法

反向合成清洗的核心思想是：如果一组 gold 三元组是高质量的，那么应该能反向生成一段清晰文本，并且这段反向文本和原始文本应该支持同一组三元组。

流程：

1. 输入 gold 三元组。
2. 用 LLM 反向生成一段干净医疗文本。
3. 比较反向文本和原始文本是否支持同一组三元组。
4. 让 teacher 模型输出 confidence 和理由。
5. confidence 低于 0.8 的样本从训练集删除或降权。

示例：

```text
Gold triple:
("儿童肥胖", "病因", "多吃少动")

反向生成文本:
儿童肥胖的发生与多吃少动密切相关。

原始文本:
儿童肥胖与生活方式、遗传等因素相关，部分患儿存在饮食结构不合理和运动不足。

判断:
原始文本能支持该三元组，confidence=0.91，保留。
```

低质量样本示例：

```text
Gold triple:
("慢性胃炎", "病因", "幽门螺杆菌感染")

原始文本:
慢性胃炎患者常出现上腹部不适、反酸等症状。

判断:
原文只描述症状，没有提到幽门螺杆菌感染，confidence=0.42，删除或降权。
```

### 7.2 为什么用反向合成清洗

为什么不用直接让 LLM 重标全文：

- 重标全文成本高，而且会引入新的标注口径。
- LLM 开放式重标可能生成 schema 外关系，后续还要再归并。

为什么反向合成更适合：

- 它围绕已有 gold 三元组做一致性判断，不会完全重写数据集。
- 它能定位“gold 与原文证据不匹配”的样本。
- 它适合批量自动清洗训练集。

为什么不能清洗测试集：

- 清洗测试集会改变评测分布，指标不再可比。
- 主结果必须在固定测试集上评估。
- 被发现有标注问题的测试样本只能作为诊断集，不进入主指标。

实际使用时我只清洗训练集。Dev/Test 只做错误分析，不参与过滤和重标。

### 7.3 结果

反向合成清洗后，训练集中约 18.6% 样本被删除或降权。阈值在 Dev 集上选择，主结果仍然在固定原始 Test 集上评测：

| 方法 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: |
| Hard Negative + DPO | 44.7 | 51.8 | 48.0 |
| + 反向合成数据清洗 | 53.4 | 58.7 | 55.9 |

阈值消融：

| 清洗策略 | 删除/降权比例 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: | ---: |
| 不清洗 | 0.0 | 44.7 | 51.8 | 48.0 |
| confidence < 0.7 | 11.2 | 50.8 | 57.1 | 53.8 |
| confidence < 0.8 | 18.6 | 53.4 | 58.7 | 55.9 |
| confidence < 0.9 | 31.4 | 54.2 | 54.8 | 54.5 |

提升原因：

- 删除了文本和三元组弱对齐样本。
- 减少了“漏标当负例”的噪声。
- 模型学到的证据边界更清晰。

## 8. 第五阶段：本体建模清洗 Schema

### 8.1 为什么需要清洗 Schema

清洗数据后，仍然有一类错误来自 schema 本身：

- 关系名语义重叠，如“病因”“诱因”“危险因素”“发病机制”。
- 关系粒度不统一，如“治疗方法”和“药物治疗”。
- 缺少 head-tail 类型约束，导致方向错误或关系误用。

如果 schema 本身不清楚，模型很难稳定地区分近义关系。

### 8.2 本体建模流程

我用了一个轻量本体建模流程来清洗 schema：

1. **开放式抽取五元组**  
   对训练集文本先不加 schema 约束，让 teacher 模型抽取开放五元组：
   ```text
   head, head_type, relation, tail, tail_type
   ```
   示例：
   ```text
   原文：高钾血症常见于肾小球滤过功能降低，导致钾排泄减少。
   五元组：高钾血症/疾病, 病因, 肾小球滤过功能降低/病理状态
   五元组：肾小球滤过功能降低/病理状态, 发病机制, 钾排泄减少/生理过程
   ```

2. **关系归一化和聚类**  
   对开放抽取到的 relation 做标准化，包括同义词归一、短语去噪和 embedding 聚类。聚类后用 LLM + 人工抽检判断哪些关系应该合并，哪些只是表面相似但语义不同。

   ```text
   引起/导致/诱发 -> 病因
   用药/药物治疗/服用药物 -> 药物治疗
   伴随/并发/合并 -> 需要区分并发症与伴随症状，不能直接合并
   ```

3. **关系定义重写**  
   给每个 relation 补充自然语言定义、正例、反例和易混关系边界。

   ```text
   病因：导致疾病发生的直接原因或明确致病因素。
   反例：疾病发生后的病理过程不标为病因，应标为发病机制。
   易混：危险因素强调风险增加，不一定直接导致疾病。
   ```

4. **head-tail 类型规则总结**  
   统计每类 relation 常见的 head_type 和 tail_type，保留高频且医学上合理的类型组合，例如：
   ```text
   病因: head_type=疾病, tail_type=病因/因素/感染
   症状: head_type=疾病, tail_type=症状/体征
   药物治疗: head_type=疾病, tail_type=药物
   ```
   类型规则不直接替代模型输出，而是用于两件事：训练时把关系定义和类型约束放入 prompt；推理后对明显非法的边降置信或过滤。

5. **轻量公理和一致性规则发现**  
   对部分关系总结方向性、互斥、传递和逆关系规则。这一步主要服务下游 Graph RAG 图谱一致性，不把推理出的新边计入抽取主指标。

   ```text
   位于/属于 类关系可以用于图谱路径推理。
   并发症 与 治疗方法 互斥，不能由同一 tail 同时承担。
   药物治疗 的 tail 类型应为药物，不应是症状或检查项目。
   ```

6. **Schema 重构和训练回灌**  
   先删除明显重复关系，再拆分粒度过粗关系，并补充业务需要但原 schema 缺失的关系。重构后的 schema 从原始 40 个关系调整为 55 个关系。训练时每个候选 relation 不只给名字，还给定义和 head-tail 类型约束：

   ```json
   {
     "relation": "药物治疗",
     "definition": "疾病或症状对应的药物干预方案",
     "head_type": ["疾病", "症状"],
     "tail_type": ["药物"],
     "confusable": ["治疗方法", "辅助治疗"]
   }
   ```

### 8.3 为什么用本体建模

为什么不直接沿用公开数据集 schema：

- 公开 schema 不一定符合医院 Graph RAG 场景。
- 部分关系边界不清晰，会放大模型混淆。
- 缺少类型约束，不利于图谱后续使用。

为什么不用纯人工改 schema：

- 人工能保证质量，但缺少数据支撑，容易凭经验拍脑袋。
- 本体建模先从开放抽取结果统计真实关系分布，再人工/LLM 辅助归并，更有依据。

为什么不用完全开放 schema：

- 开放 schema 会导致关系爆炸。
- Graph RAG 需要稳定图谱结构，不能每批文档生成一套关系名。

### 8.4 结果

Schema 从原始 40 个关系重构为 55 个关系，并为核心关系补充了 head-tail 类型约束和关系定义。

重新训练并评测：

| 方法 | Precision | Recall | F1 |
| --- | ---: | ---: | ---: |
| + 反向合成数据清洗 | 53.4 | 58.7 | 55.9 |
| + relation definition | 56.4 | 60.1 | 58.2 |
| + head-tail 类型约束 | 58.2 | 63.1 | 60.6 |
| + 关系簇重采样 Hard Negative | 59.8 | 65.2 | 62.4 |

错误类型变化：

| 错误类型 | 数据清洗后 | Schema 清洗后 |
| --- | ---: | ---: |
| 关系混淆 | 22.6 | 15.3 |
| Head/Tail 类型非法 | 13.8 | 6.4 |
| 无证据过抽取 | 18.1 | 13.7 |
| 实体边界错误 | 15.6 | 13.9 |

## 9. 端到端 Graph RAG 评测

离线抽取指标提升不够，还需要验证下游 Graph RAG 是否受益。

评测流程：

1. 用不同版本抽取模型构建医疗知识图谱。
2. 对同一批医疗 QA 问题执行 Graph RAG。
3. 比较 Evidence Recall、QA Accuracy、错误证据率。

结果：

| 图谱构建方案 | 图谱边数 | 错误边率 | Evidence Recall | QA Accuracy |
| --- | ---: | ---: | ---: | ---: |
| 原始 SFT | 41,280 | 18.7 | 61.4 | 54.8 |
| + Hard Negative + DPO | 33,180 | 10.8 | 66.2 | 62.1 |
| + 数据清洗 + Schema 清洗 | 29,640 | 7.8 | 70.4 | 66.2 |

图谱边数下降不是坏事，说明过度抽取减少；错误边率下降后，下游 Graph RAG 检索到的证据更集中。

## 10. 指标可信度与保守口径

从面试可信度看，原来把最终 Triple-F1 写到 68.1 偏激进，除非能明确证明它是在固定原始 Test 集上得到的，并且没有清洗或重标 Test。更稳妥的主口径是：

- 固定 CMEIA 原始 Test 集：Triple-F1 `35.1 -> 62.4`。
- 下游 Graph RAG QA Accuracy：`54.8 -> 66.2`，提升 11.4 个百分点。
- 错误边率：`18.7% -> 7.8%`。

如果内部诊断集或清洗后评测集上能到 66-68 F1，可以作为诊断集结果讲，但不要放在主结论里。面试官最容易追问的是“你是不是清洗了测试集”和“schema 重构后 gold relation 怎么映射”，所以主文档必须坚持：Test 固定，清洗只作用于训练集，schema 映射规则在 Dev 上确定后再跑 Test。

## 11. 面试表达

### 11.1 30 秒概述

这个项目是为医疗 Graph RAG 做 schema 约束三元组抽取。我的推进路径是先基于 CMEIA 数据构造 SFT 指令数据，训练小模型替代 DeepSeek-R1；第一版发现过度抽取明显，于是做负 schema 扩增和关系簇 Hard Negative；随后针对“儿童肥胖”“持续上腹阵痛”这类医学边界偏好问题，用 DPO 做 chosen/rejected 对齐。模型侧优化后 precision 仍然不高，我判断瓶颈在数据和 schema，于是只对训练集做反向合成清洗，再用本体建模方法重构 relation 定义和 head-tail 类型约束。固定 CMEIA Test 集上 Triple-F1 从 35.1 提升到 62.4，下游 Graph RAG QA 准确率提升 11.4 个百分点。

### 11.2 2 分钟深挖讲法

第一步是 SFT 冷启动。我把 CMEIA 原始数据改造成 schema 约束指令数据：输入是文本和候选 schema，输出是按 schema 分组的三元组，没有证据的关系输出空数组。第一版模型格式学得不错，但 precision 只有 30 左右，badcase 主要是过度抽取。

针对过度抽取，我先做负 schema 扩增。因为模型看到医学实体和候选关系后，会利用医学常识补全关系，所以需要大量“候选关系存在但原文无证据”的训练样本。随后我发现随机负 schema 太简单，对病因/发病机制这类近义关系帮助有限，于是把易混淆关系放到同一个候选 schema 中做 Hard Negative。

第二步是 DPO。很多错误不是完全不会抽，而是输出偏好不符合医学场景。比如模型把“儿童肥胖”抽成“肥胖”，把“持续上腹阵痛”抽成“上腹阵痛”。这类问题用 SFT 只能告诉模型正确答案，但 DPO 可以直接让 chosen 相对 rejected 概率更高，所以更适合实体边界和修饰语保留。

第三步我发现模型侧方法做完后，precision 还是上不去，于是转向数据质量。CMEIA 里有些长文本对应很少三元组，标注不完整或证据弱。我的做法是反向合成：用 gold 三元组反向生成干净文本，再比较原文是否支持同一组三元组，低置信样本删除或降权。

最后是 schema 清洗。CMEIA 原始 schema 存在关系重叠和粒度不一致，所以我用轻量本体建模：先开放式抽五元组，再聚类归并关系，给每个 relation 写定义、正反例和易混边界，再统计 head-tail 类型规则。重构后的 schema 会回灌到训练 prompt 和 hard negative 采样中，而不是只做后处理。

### 11.3 面试官追问回答

Q：为什么第一步用 SFT？

A：SFT 解决的是任务格式和基础 schema 跟随能力。如果没有 SFT，后面的 DPO 没有稳定 reference，模型连按 schema 输出空数组都不稳定。

Q：为什么要做负 schema？

A：因为过度抽取本质是拒抽能力不足。模型看到候选关系不代表一定要抽，负 schema 能明确训练“无证据输出空”。

Q：为什么 Hard Negative 比随机负采样好？

A：随机负关系通常和原文无关，模型很容易输出空；真正难的是病因/发病机制这类近义关系。Hard Negative 把这些易混淆关系放在一起，强迫模型学关系边界。

Q：为什么用 DPO？

A：儿童肥胖、持续上腹阵痛这类问题是偏好问题，不是完全不会抽。DPO 能利用 chosen/rejected 对比，明确降低边界不完整输出的概率。

Q：为什么后面转向数据清洗？

A：因为模型侧优化后 precision 仍然上不去，说明噪声来自训练数据和 schema。继续堆训练技巧收益低，必须处理 text-triple 弱对齐和 schema 语义重叠。

Q：为什么用反向合成清洗？

A：直接重标全文成本高，且会引入新口径。反向合成围绕已有 gold 三元组判断原文是否支持，能批量定位低质样本。

Q：为什么做本体建模清洗 schema？

A：schema 是模型输出空间。如果关系本身边界不清，模型再强也会混淆。本体建模能把关系归并、定义和 head-tail 类型规则补齐，让抽取和下游图谱都更稳定。

Q：你有没有清洗测试集？

A：主结果没有清洗测试集。反向合成只用于训练集删除或降权低置信样本，Dev/Test 只做错误分析。固定 Test 集保持原始标注，避免评测口径被污染。

Q：为什么最终 F1 不写 68？

A：68 左右更适合作为清洗诊断集或内部重标口径，面试主口径我会用固定原始 Test 集的 62.4。这个口径更稳，也更容易解释没有测试集泄漏。

## 12. 简历表述

面向医疗 Graph RAG 构建 Schema 约束三元组抽取流程，基于 CMEIA 医疗三元组数据构造 1.4w 条指令样本进行 SFT 冷启动；针对第一版模型过度抽取问题，引入负 schema 扩增与关系簇 Hard Negative，提升无证据拒抽和病因/发病机制等近义关系区分能力；针对“儿童肥胖”“持续上腹阵痛”等医学实体边界与修饰语偏好问题，基于 badcase 构造 DPO chosen/rejected 偏好对；在模型侧优化后进一步通过反向合成清洗训练集 text-triple 弱对齐样本，并用本体建模方法重构 relation 定义、易混关系簇与 head-tail 类型约束。固定 CMEIA Test 集上 Triple-F1 由 35.1 提升至 62.4，错误边率由 18.7\% 降至 7.8\%，下游 Graph RAG QA 准确率提升 11.4 个百分点。
