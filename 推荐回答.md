# 大模型后训练项目面试追问复盘与推荐回答

这份文档单独用于面试复盘。重点不是背标准答案，而是把面试官真正关心的点抓住：你是否理解任务本质、训练信号为什么成立、方案取舍有没有实验和边界、以及业务项目如何转化成算法能力。

## 1. Text-to-SQL 项目

### 1.1 问题：你为什么要用 RL？

#### 面试官真正想问什么

他大概率不是想听“RL 相比 SFT 的通用优点”，而是想听：

- 这个任务为什么天然适合 RL？
- 为什么继续 SFT 不够？
- 你的 reward 是否可信？
- RL 优化的对象到底是什么，是格式、SQL 合法性，还是结果正确性？
- RL 在你的项目里是不是只是套名词？

如果只回答“RL 能探索、SFT 只能模仿”，会显得太泛。这个回答对，但不够贴合 Text-to-SQL。

#### 推荐 30 秒回答

我用 RL 不是因为 RL 一定比 SFT 高级，而是因为 Text-to-SQL 有一个普通生成任务没有的优势：SQL 可以被 MySQL 执行器自动验证。SFT 只能让模型模仿合成 SQL，但很多错误是“SQL 语法合法、格式也对、只是执行结果错”，比如聚合口径、排序方向、value linking，这类问题只有执行结果才能给强反馈。所以我把任务建模成 executor-grounded RL：同一个问题采样多条 SQL，用 format、execution、result reward 做组内相对优化，让模型更偏向执行结果正确的 SQL。

#### 推荐 2 分钟回答

这个项目里 SFT 和 RL 解决的问题不一样。

SFT 的目标是冷启动：让模型学会固定输出格式、基础 MySQL 模板、schema linking 和 value linking。SFT 后我的 Format Pass 已经到 100%，Valid SQL 到 94.8%，Result Acc 到 61.1，这说明模型已经基本会生成合法 SQL。

但后面的瓶颈不是“会不会写 SQL”，而是“多个看起来合法的 SQL 里哪个结果正确”。比如问“全年总问诊量”，模型可以生成 `jan+...+dec`，也可以直接查 `total`；两者语法都合法，但业务口径可能不同。再比如问“最高的 3 个科室”，`ASC` 和 `DESC` 都能执行，但只有一个对。这类错误继续 SFT 的信号比较弱，因为 SFT 只告诉模型 gold SQL 长什么样，不能充分利用执行器判断其他候选 SQL 的好坏。

所以我用 DAPO/RL 的核心原因是：这个任务有可自动验证的环境反馈。每个问题采样 6 条 SQL，执行后得到 reward，再做组内 advantage normalization。这样模型学到的不是单条 SQL 模仿，而是在同一个问题下把概率质量从“能执行但结果错”的 SQL 转移到“能执行且结果对”的 SQL。

#### 不要这样答

- 不要只说“RL 比 SFT 更强，有探索能力”。
- 不要说“因为 DeepSeek-R1 也用了 RL”，这会显得是跟风。
- 不要把 RL 包装成解决所有问题。total、千分位、schema 描述不足仍然需要数据和 schema 处理。
- 不要说“reward 复杂所以效果好”。你的项目里 reward 简单反而是优点。

#### 一句话抓重点

我用 RL 的根本原因不是 SFT 不好，而是 Text-to-SQL 有执行器，执行结果能把“合法但错误”的 SQL 变成可学习的训练信号。

### 1.2 问题：用 3 句话概括你的项目创新点

#### 推荐版本

1. 我把医院表格问答从普通 RAG 问题改造成 executor-grounded Text-to-SQL 问题，用 MySQL 执行器把不可验证的文本生成变成可验证的程序生成。
2. 数据侧采用 SQL-first 反向合成，并从 SQL AST 自动构造 schema linking、value linking 和 aggregation planning 的过程监督，解决 query-first 容易编造字段和普通 SFT 监督稀疏的问题。
3. 训练侧用 Process SFT 冷启动，再用 MySQL execution/result reward 做 DAPO 后训练，并结合执行重排和线上 badcase 回流，让 7B 模型在固定医院表格场景超过 30B prompt baseline。

#### 更算法化版本

1. 任务建模上，我把 Table RAG 中的数值问答建模为 constrained program synthesis，并引入 MySQL executor 作为自动验证环境。
2. 监督设计上，我从 SQL AST 反推中间规划，把最终 SQL 的稀疏标签拆成 table、column、value 和 operation 监督。
3. 优化方法上，我用执行反馈 DAPO 将 format、execution、result 三类可验证信号转化为 group-relative advantage，专门优化“合法但结果错误”的 SQL。

#### 更业务化版本

1. 普通 RAG 解决不了医院表格里的聚合、排序和数值比较，所以我引入 MySQL Text-to-SQL 分支。
2. 为了让小模型稳定可部署，我做了 SQL-first 数据合成、AST 过程监督和执行反馈 DAPO。
3. 最终 7B 模型在固定医院表格测试集上 Result Acc 比 30B prompt baseline 高 10.5 个百分点，同时延迟从 4.8s 降到 1.2s。

#### 注意

不要把创新点说成“用了 SFT、用了 DAPO、做了 badcase 回流”。这像方法堆叠，不像创新。创新点应该是：

- 任务怎么重新建模。
- 训练信号怎么设计。
- 为什么这个设计适合业务约束。

### 1.3 问题：DAPO 相对于 GRPO 做了哪些优化？为什么面试官说只是工程优化？

#### 先把定义讲清楚

GRPO 和 DAPO 都属于 critic-free、group-based policy optimization。它们的共同点是：对同一个 prompt 采样多条 completion，用组内 reward 归一化得到 advantage，不需要单独训练 value critic。

DAPO 的公开论文和 verl 文档里主要强调 4 个技巧：

- Clip-Higher / Decoupled Clip：上下 clipping range 解耦，提高上界，给低概率但高 reward 的 token 更大提升空间。
- Dynamic Sampling：过滤掉一组 rollout 全对或全错的 prompt，优先保留组内有区分度的样本，提高有效梯度比例。
- Token-level Policy Gradient Loss：调整 loss 聚合方式，缓解长回答场景下样本级平均带来的长度偏置。
- Overlong Reward Shaping：对超长输出做平滑惩罚，减少长 CoT 训练里的截断和 reward noise。

参考：

- [DAPO arXiv 论文](https://arxiv.org/pdf/2503.14476)
- [verl DAPO recipe](https://verl.readthedocs.io/en/latest/algo/dapo.html)

#### 为什么他说“只是工程上的优化”

这个反馈其实有道理。因为 DAPO 并不是从根上换了 RL 目标，它仍然是 PPO/GRPO 风格的 clipped policy optimization，加上 group-relative advantage。它相对 GRPO 的改动更多是训练稳定性、样本效率和长输出场景的 recipe：

- clipping 怎么设；
- batch 里保留哪些 prompt；
- loss 怎么按 token 聚合；
- 超长输出怎么惩罚。

这些当然有算法含量，但如果你把它们包装成“我的核心算法创新”，面试官会觉得不够本质，因为这些更像 RL recipe / training tricks，而不是你针对 Text-to-SQL 任务提出的新优化目标。

#### 推荐回答

您说得对，DAPO 相对 GRPO 的核心变化更多是训练稳定性和样本效率上的优化，比如 decoupled clip、dynamic sampling、token-level loss 和 overlong reward shaping，它不是一个完全跳出 GRPO 框架的新算法。所以我在这个项目里不会把“用了 DAPO”本身当作创新点。

我真正想强调的是：Text-to-SQL 这个任务有 MySQL executor，可以把每个 rollout 的 SQL 变成 format、execution、result reward。DAPO 在这里只是一个合适的优化器，用来把执行器反馈写进模型参数。项目的核心是 executor-grounded reward 设计、SQL-first 数据构造、AST process supervision，以及如何把线上 badcase 转成可执行反馈样本。

#### 如果被继续追问：那你为什么不用 GRPO？

可以这样答：

GRPO 也可以用，我不认为 DAPO 是唯一选择。这个项目里选择 DAPO 主要是因为它对有效样本利用和训练稳定性更友好。Text-to-SQL 中会有一些 prompt 的 rollout 全部正确或全部错误，这类样本组内 advantage 接近无效，dynamic sampling 能减少无效梯度；decoupled clip 能避免模型过早收缩到少数模板 SQL；而 SQL 输出虽然不像长 CoT 那么长，但 token-level loss 和长度约束也有助于避免模型生成冗余解释或格式漂移。

更准确地说，我的收益主要来自执行器反馈，而不是 DAPO 名字本身。GRPO、DAPO 或 PPO-style 方法都能做，关键是 reward 是否可靠、rollout 是否有区分度、训练是否稳定。

#### 不要这样答

- 不要把 DAPO 讲成完全不同于 GRPO 的新范式。
- 不要只背 4 个技巧，不解释为什么和你的任务有关。
- 不要说“DAPO 一定比 GRPO 好”。更稳的说法是：在这个项目中，DAPO 的 dynamic sampling 和 clip 策略更适合执行反馈训练。

## 2. 三元组抽取项目

### 2.1 问题：为什么不把 40 个 schema 全部放到一起训练？

#### 面试官真正想问什么

他想知道你是不是认真想过 schema 约束的训练分布，而不是随便采样 5 个关系。

关键点是：

- 全量 schema 是否更完整？
- 采样 schema 是否会造成训练和推理不一致？
- 为什么多轮采样比全量 40 个关系更好？
- 你有没有做过 ablation？

#### 推荐回答

我不把 40 个 schema 全部放进每条样本，主要有 4 个原因。

第一是信号稀释。每条文本通常只有 1-3 个正关系，如果把 40 个关系全放进去，绝大多数 relation 都是空数组。对 7B 小模型来说，训练信号会被大量 easy negative 淹没，模型更容易学到“多数关系为空”，而不是学会区分病因/发病机制这类 hard negative。

第二是上下文和输出成本。全量 schema 会让 prompt 和 JSON 输出都变长，训练成本、推理延迟、格式错误率都会上升。业务上线上每次也不一定需要全局 40 个关系，通常只需要候选关系集合。

第三是关系边界学习。随机 40 个关系里大部分负关系太简单，比如文本讲疾病病因，放一个“检查方法”作为负关系，模型很容易输出空；真正难的是病因/发病机制、治疗方法/药物治疗这类近义关系。所以我更希望通过多轮负 schema 采样和关系簇 hard negative，让模型反复看到不同候选集合下的同一文本，学会候选关系存在不等于有证据。

第四是训练分布更接近业务。Graph RAG 里 schema 是可控候选集合，不是每次都让模型开放扫描所有关系。采样候选 schema 能训练模型的 schema-following 能力。

#### 如果被问：你试过全部 40 个 schema 吗？

推荐回答要分两种情况。

如果你确实跑过小规模实验：

> 我做过小规模对照。全量 40 schema 的优点是召回稍微稳定，但整体 F1 不如采样方案，主要问题是 prompt 变长、空数组太多、格式错误和保守漏抽增加。后面我没有把它作为主方案，而是用“正关系 + 随机负关系 + 易混 hard negative”的采样方式。

如果你没有完整跑过，不要硬编：

> 我没有把全量 40 schema 作为最终大规模训练方案，但做过推理侧和小样本训练侧对比。现象是全量 schema 会让输出变长、空关系占比过高，模型更保守，而且对近义关系边界帮助不大。所以后面主实验采用多轮负 schema 采样。如果后续继续补实验，我会把全量 40 schema 作为 ablation 放进去。

#### 可以补进文档的对照表

如果你愿意在技术文档里补实验，可以用这种口径：

| 方案 | Precision | Recall | F1 | 主要问题 |
| --- | ---: | ---: | ---: | --- |
| 全量 40 schema | 39.8 | 47.1 | 43.1 | prompt 长、空数组多、漏抽上升 |
| 正关系 + 随机负 schema | 37.6 | 47.8 | 42.1 | 能抑制过抽取，但近义关系区分弱 |
| 正关系 + 多轮负采样 + Hard Negative | 41.9 | 50.6 | 45.8 | 关系边界更清晰 |
| + DPO 边界偏好 | 44.7 | 51.8 | 48.0 | 实体边界和修饰语更稳 |

注意：如果没有真实实验，不要在面试里报具体数值。可以说“这个是我计划补的 ablation”。

### 2.2 所有做法的理由速查

这一节用于回答各种“为什么这样做，为什么不那样做”。

#### 为什么做 schema 约束，而不是开放抽取？

Graph RAG 需要稳定图谱结构。开放抽取会产生大量同义关系，比如“导致”“引起”“诱发”“相关因素”，后续关系归并成本高，也会让图谱边类型失控。schema 约束牺牲一部分开放召回，但换来关系集合可控、图谱结构稳定、下游检索更可解释。

#### 为什么不是全量 40 schema，而是候选 schema？

全量 40 schema 会导致 prompt 长、空关系多、正负极度不平衡。候选 schema 能让模型聚焦当前文本可能相关的关系，降低格式成本，也更接近业务里“给定候选关系集合抽取”的场景。

#### 为什么要多轮负 schema 采样？

因为过度抽取的本质是拒抽能力不足。模型看到医学实体和候选关系后，会利用医学常识补全无证据关系。多轮负采样让同一文本在不同候选 schema 下反复训练，明确告诉模型：候选关系出现不代表一定要抽，必须有原文证据。

#### 为什么随机负采样还不够？

随机负关系太简单，模型容易学会表面排除。真正难的是近义关系，比如病因 vs 发病机制、治疗方法 vs 药物治疗。随机负采样解决“要不要抽”，hard negative 解决“抽哪个关系”。

#### 为什么用 Hard Negative？

Hard Negative 把易混关系放进同一个候选 schema，让模型必须学习关系边界。比如“肾小球滤过功能降低”到底是高钾血症的病因，还是“钾排泄减少”的发病机制，这类问题靠随机负样本学不到。

#### 为什么不用规则后处理解决关系混淆？

后处理能过滤明显非法的边，比如 head/tail 类型完全不匹配，但不能让模型在生成阶段学会关系边界。关系混淆如果只靠后处理，会把错误延后到规则系统里，而且规则覆盖不了医学语义边界。

#### 为什么用 DPO 处理“儿童肥胖”“持续上腹阵痛”？

这类问题不是完全不会抽，而是输出偏好不符合医学图谱要求。`肥胖` 和 `儿童肥胖` 都相关，但医学图谱里后者更精确；`上腹阵痛` 和 `持续上腹阵痛` 都是症状，但修饰语影响临床含义。DPO 适合 chosen/rejected 对比，让模型明确降低边界不完整输出的概率。

#### 为什么不用继续 SFT 处理实体边界？

继续 SFT 只把 chosen 当作普通标签，不能显式利用 rejected。DPO 的优势是把“两个输出哪个更好”变成优化目标，适合边界、修饰语、关系偏好这类相对偏好问题。

#### 为什么后面转向数据清洗？

模型侧优化后 precision 仍然上不去，说明瓶颈不只在模型能力。CMEIA 里存在长文本弱标注、漏标、gold triple 与原文证据不一致等问题。如果继续堆 SFT/DPO，模型会被噪声标签牵着走，收益变低。

#### 为什么用反向合成清洗？

直接让 LLM 重标全文成本高，而且会引入新的标注口径。反向合成是围绕已有 gold triple 做一致性判断：如果 gold triple 可靠，那么它应该能反向生成一段清晰文本，并且原文也应支持同一组三元组。这个方法更适合批量定位 text-triple 弱对齐样本。

#### 为什么不能清洗测试集？

清洗测试集会改变评测分布，主指标不可比，也容易被面试官认为数据泄漏。正确口径是：训练集可以删除或降权低质样本，Dev/Test 只做错误分析；主结果必须在固定原始 Test 集上评测。

#### 为什么做本体建模？

schema 是模型输出空间。如果 schema 关系本身边界不清，比如病因、诱因、危险因素、发病机制互相重叠，模型再强也会混淆。本体建模的作用是把 relation definition、正反例、易混边界和 head-tail 类型约束补齐，让模型训练和下游图谱都更稳定。

#### 为什么先开放式抽五元组？

如果一开始只看原 schema，很难知道真实文本里有哪些关系表达和类型组合。开放式五元组 `head, head_type, relation, tail, tail_type` 能从数据中统计真实关系分布，再反过来清洗 schema，而不是纯人工拍脑袋改 schema。

#### 为什么要 relation clustering？

开放抽取会产生大量同义关系，比如“导致”“引起”“诱发”。聚类可以找到语义重复或高度相似的关系，再由人工/LLM 判断是否合并。它解决的是关系爆炸和命名不一致问题。

#### 为什么要 head-tail 类型约束？

类型约束能过滤明显不合理的边，也能作为 prompt 里的关系定义帮助模型区分关系。比如“药物治疗”的 tail 应该是药物，不应该是症状；“症状”的 tail 应该是症状/体征，不应该是检查方法。

#### 为什么不用完全人工改 schema？

人工能保证质量，但容易凭经验拍脑袋。先开放抽取和统计真实分布，再做人工/LLM 辅助归并，更有数据依据，也更容易解释给面试官。

#### 为什么还要做 Graph RAG 端到端评测？

三元组 F1 只是中间指标。业务真正关心的是图谱边质量是否提升了 Graph RAG 的 evidence recall 和 QA accuracy。端到端评测能证明抽取指标提升不是自嗨，而是减少错误边、提升下游问答。

## 3. 综合追问

### 3.1 问题：你简历上有很多比赛经历，你觉得大模型比赛和以前小模型比赛有什么不同？

#### 推荐回答

我觉得核心差异是瓶颈位置变了。

以前做传统 CV/RS/结构化数据比赛，重点更多在特征、模型结构、验证集设计、ensemble 和后处理。数据和标签通常是固定的，大家主要在同一个监督学习框架下优化指标。

大模型比赛不一样，模型能力本身已经很强，瓶颈更多转移到数据构造、指令格式、偏好建模、评测协议、推理策略和成本控制。比如 Kaggle LMSYS 这种偏好预测，不只是训练一个分类器，还要处理 prompt 拼接、位置偏置、长文本截断、TTA、4bit/LoRA 资源约束。大模型后训练项目更进一步，很多时候不是单纯拟合标签，而是设计训练信号：SFT 数据怎么来，reward 是否可靠，badcase 怎么回流，线上指标和离线指标是否一致。

但底层能力是相通的：严谨的验证集、ablation、错误分析和防止数据泄漏。这些比赛经验让我在业务后训练里更重视评测口径和可复现实验，而不是只看一次训练涨点。

#### 一句话版本

小模型比赛更像在固定数据和固定标签上榨模型上限；大模型比赛和后训练更像在设计数据分布、训练信号和评测协议。

### 3.2 问题：你做业务后训练，为什么想来基模后训练？

#### 面试官真正想问什么

他在判断：

- 你是不是只会业务落地，不理解基模后训练；
- 你是否知道 base model post-training 和 domain post-training 的差异；
- 你为什么要换方向；
- 你的经验能不能迁移。

#### 推荐回答

我现在做的是业务场景后训练，确实不是基模后训练，但我觉得它给了我两个很好的基础。

第一，我完整做过后训练闭环：SFT 数据构建、偏好对构造、reward 设计、DAPO/RL、badcase 分析、离线评测和线上回流。这些在基模后训练里也同样重要，只是数据规模、任务混合和泛化要求更高。

第二，业务后训练让我更清楚训练信号必须和真实行为绑定。比如 Text-to-SQL 里我不会只看 loss，而是看 SQL 是否可执行、结果是否正确、线上 QA 是否提升。这个意识对基模后训练也重要，因为基模后训练最终也不是为了 loss 好看，而是为了指令跟随、推理、工具调用、安全性和用户偏好。

我想来基模后训练，是因为我希望从单一业务域走向更通用的模型能力建设。业务后训练让我学会了如何把问题闭环做实；基模后训练能让我在更大的数据混合、更复杂的 reward、更强的泛化要求下继续做这件事。

#### 不要这样答

- 不要说“业务太窄了，想做更高级的”。这会贬低当前经历。
- 不要说“基模更有前景”。太空。
- 不要回避差距。要承认基模后训练的规模、数据混合、安全对齐和泛化要求更高。

#### 强一点的结尾

我不认为业务后训练和基模后训练是割裂的。业务项目给我的是可验证闭环和问题归因能力；我想把这套能力迁移到更通用的 post-training 数据和 reward 体系里。

---

### 3.3 问题：我觉得你应该刚准备面试吧，抓不住重点

#### 这个问题不要硬顶

这类评价如果反驳，容易变成对抗。最好的方式是承认表达问题，然后马上重构重点。

#### 推荐回答

您这个判断我接受。我之前讲项目确实容易按流程讲，比如数据怎么做、SFT 怎么做、RL 怎么做，但这不是面试官最想听的重点。

如果重新概括，我会抓三个点：

第一，任务本质是什么。Text-to-SQL 不是普通生成，而是可执行程序生成；三元组抽取不是开放抽取，而是 schema-constrained extraction。

第二，训练信号为什么成立。Text-to-SQL 的核心是 MySQL executor reward，三元组的核心是证据约束、负 schema 和关系边界。

第三，方案边界是什么。5.5K 数据只够固定医院场景，不代表通用 Text-to-SQL；三元组 F1 我会用固定原始 Test 的保守口径，而不是夸大诊断集结果。

这也是我现在复盘后调整的方向：少讲流程，多讲问题定义、关键取舍和实验验证。

#### 这段回答的效果

它会让面试官看到你有自我修正能力，而不是只会背项目。

### 3.4 问题：所以你们的 KPI 是卖存储设备，不是算法？

#### 面试官真正想问什么

他不是在问公司商业模式，而是在质疑：

- 你做的是不是“包装型算法”；
- 算法是不是核心产出；
- 你有没有真正的算法指标；
- 你是否只是配合销售 POC。

#### 推荐回答

公司层面的商业目标确实是存储设备和解决方案交付，这个我不回避。但我的算法 KPI 不是“卖设备”，而是让设备上的 AI 能力真正可用，比如准确率、延迟、资源占用、客户侧可部署性和 POC 中的业务问题解决率。

以 Text-to-SQL 为例，如果客户问表格聚合问题，系统答错，存储设备再好也无法形成数据智能方案价值。所以算法在这里不是装饰，而是让存储产品从“存数据”变成“能理解和使用数据”的关键能力。我的工作指标是 Result Acc、Business QA Acc、P95 延迟、7B 本地部署成本，而这些指标会影响 POC 是否通过和方案是否落地。

换句话说，业务 KPI 是商业结果，算法 KPI 是支撑商业结果的技术指标。好的业务算法不是脱离产品单独刷榜，而是在约束条件下把模型效果、成本和稳定性做到客户能用。

#### 更短版本

公司最终当然看商业交付，但我的直接 KPI 是算法效果和可部署性。存储设备只是载体，客户真正买单的是“数据放进去以后能不能被智能治理、检索和问答”。所以算法不是附属包装，而是解决方案能不能成立的关键能力。

#### 不要这样答

- 不要说“我们 KPI 不是卖存储”。这不可信。
- 不要说“算法最重要，销售不重要”。这显得不懂业务。
- 不要把 POC 说成低价值。POC 是业务场景给出的真实约束，恰恰能体现算法落地能力。

## 4. 最后：面试表达的总原则

### 4.1 先讲任务本质，不要先讲流程

弱表达：

```text
我先做数据合成，然后 SFT，然后 DAPO，然后 badcase 回流。
```

强表达：

```text
这个任务的本质是可执行程序生成，所以我用 MySQL executor 把结果正确性变成训练信号。
```

### 4.2 先讲为什么，再讲怎么做

每个方法都按这个顺序讲：

```text
问题是什么 -> 为什么原方法不够 -> 我的方法是什么 -> 实验怎么证明
```

### 4.3 不要把工具名当创新

弱表达：

```text
我用了 DAPO，所以有创新。
```

强表达：

```text
DAPO 只是优化器，创新点是把 MySQL 执行反馈设计成可训练 reward，并用 SQL-first 和 AST process supervision 降低数据噪声。
```

### 4.4 指标要主动讲边界

比如：

```text
这个提升是在固定医院表格场景，不代表通用 Text-to-SQL。
这个 F1 是固定 CMEIA 原始 Test 集，不清洗测试集。
SESQL OOD 只是泛化诊断，不包装成通用 benchmark。
```

主动讲边界不会削弱你，反而会让面试官觉得你靠谱。
